# Diskwise 设计词汇表

> 团队共享概念定义，确保讨论时用词一致。
> 每个概念附带"来源"说明它是怎么进入设计的。

---

## Scaffold（脚手架）

**定义：** 某个软件的"可清理内容"定义。比如 "WeChat 的图片缓存可以删、视频缓存可以删，但聊天数据库不能碰"。它是一个**逻辑概念**，不特指某种实现方式。

**来源：** 原始需求来自 "per-app scaffolds (like clean-my-wechat)" — 用户想要每个知名软件有一个清理方案。CleanMyWechat 是一个 Python 脚本，里面硬编码了 WeChat 清理逻辑。

**当前实现：** 每个 scaffold 是一个 `.toml` 文件，声明了检测路径、清理范围、风险等级。

---

## TOML 模式

**定义：** 当前采用的实现方式。每个软件对应一个 `.toml` 文件，系统加载后做：读文件 → 匹配路径 → 执行操作。

```
[detect]
paths = ["%APPDATA%/Tencent/WeChat/**/FileStorage"]

[[scope]]
id = "image-cache"
glob = "FileStorage/Image/**"
risk = "low"
```

**特征：** 声明式。你说"这是什么"，系统照做，不问为什么。

**优点：** 社区贡献门槛低、与代码解耦、容易验证。
**缺点：** 静态路径会随软件更新失效、无法表达版本差异（如 WeChat 3.8 vs 3.9 路径不同）、风险等级对所有用户一刀切。

**来源：** Claude 在设计阶段将"per-app scaffolds"这个需求翻译为了 TOML 文件格式。这是一个实现层面的选择，并非经过深入推敲的架构决策。

---

## 管线模式（Pipeline）

**定义：** 将清理流程拆分为多个独立阶段，每个阶段各司其职：

```
检测 → 分析 → 评估 → 执行 → 验证
```

不是"读 TOML → 删文件"的两步走，而是系统自己走完完整的判断链路。

**与 TOML 的关系：** 不是替换关系，是升级通道。

- 80% 的已知软件（npm/pip/cargo/Chrome 等），TOML 就够用。这些软件结构稳定，路径十几年不变。
- 那 20% 复杂软件（WeChat、Steam、Docker 等），如果发现 TOML 不够用（版本差异、动态 scope），才升级到 detector。
- **用户看到的完全一样**——不管底层是 TOML 还是 detector，界面都显示"可清理 4.2 GB"。

**来源：** 借鉴 Claude Code 泄漏代码中 harness 的设计思想——其查询引擎不是直接调 API，而是经过上下文压缩 → 权限检查 → 工具路由 → 流式缓存的管线。

---

## Harness（马具/基础设施层）

**定义：** 围绕核心能力构建的基础设施层。不是模型本身变聪明了，而是这层基础设施让模型能更好地发挥作用。

**Claude Code 的例子：** 其 46,000 行的查询引擎（harness）做了 4 件事：
1. 上下文压缩（9 段压缩，把有用信息提炼出来）
2. 权限检查（按权限级别过滤可用工具）
3. 工具路由（解析工具调用并分发到对应 handler）
4. 流式缓存（流式响应的缓存策略）

**对应到 Diskwise：** 未来可能的 harness 不是某个 scaffold 具体怎么清理，而是**整个检测-分析-执行的框架本身**。Steam 的清理逻辑可能只有 50 行，但让这 50 行能正常工作的管线框架是基础设施。

**护城河意义：** 别人抄走你的 Steam 清理脚本很容易，但抄走整个管线框架 + 多年积累的检测指纹 + 边界判断规则，就要花大功夫。TOML 文件本身没有护城河。

**来源：** 从 Claude Code 泄漏源码中得到启发——真正的智能不来自于模型本身有多强，而来自于围绕模型构建的那层基础设施。

---

## 扫描器（Scanner）

**定义：** 扫描磁盘、返回文件树的模块。

**当前实现：** walkdir（遍历文件系统，较慢但跨平台）
**计划升级：** 直读 NTFS MFT（Windows 专属，WizTree 速度的来源）

---

## AI 顾问（Advisor）

**定义：** 对未知文件夹（没有匹配 scaffold 的）调用 AI 做判断。只发送路径元数据（大小、文件数、扩展名分布、抽样文件名），不发送文件内容。

**默认提供者：** Ollama（本地运行，无需联网）
**可选提供者：** OpenAI / Anthropic

---

## 执行器（Executor）

**定义：** 实际执行删除操作的模块。

**安全策略（由安全到危险）：**
1. 回收站（默认）—— 用户可以在资源管理器还原
2. 隔离区（可选）—— 移到指定目录保留 7 天
3. 直接删除（危险）—— 仅限用户明确确认的场景

**所有操作记录 JSONL 撤销日志。**

---

## 拐点

**定义：** 需要从 TOML 升级为 detector 的那个时机。

**含义：** TOML 和 detector 不是二选一，而是递进关系。当一个软件的清理逻辑复杂到：
- TOML 无法表达版本差异
- 社区反复报"路径找不到"
- 需要判断运行时状态（进程是否在跑）

这时候就该写 detector 了。

**估算：** 不是按总量算的。WeChat/Steam/Docker 可能在第 1 天就需要 detector，npm/pip 可能永远不需要。
